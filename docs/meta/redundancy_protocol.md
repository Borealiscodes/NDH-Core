# Redundancy Protocol (UMA Meta Forest)
The Redundancy Protocol defines how UMA identifies, evaluates, and resolves
duplicate or overlapping documents across the multiple planes of the Meta Forest.
It ensures coherence, prevents index sprawl, and maintains structural clarity
as the forest grows.

---

## 1. Purpose
Redundancy is natural in a multi-plane system. This protocol exists to:
- detect when two documents serve the same structural role
- distinguish true duplicates from parallel-plane variants
- define how to merge, move, or deprecate redundant files
- prevent multi-index drift in the meta layer
- maintain conceptual clarity across the forest

Redundancy is not an error; it is a signal.

---

## 2. Redundancy Categories
UMA recognizes three types of redundancy:

### A. Structural Redundancy
Two documents attempt to perform the same *meta* function.
Examples:
- multiple meta indexes
- multiple growth rulebooks
- multiple navigation roots

These must be resolved immediately.

### B. Plane Redundancy
Two documents mirror each other across different planes.
Examples:
- a mythic version and a structural version
- a cartographic version and a meta version

These may coexist if clearly labeled.

### C. Narrative Redundancy
A document echoes another through story, metaphor, or interface.
Examples:
- Pip’s guide vs. cartographer’s guide
- clearing vs. memorial reflections

These are allowed and often intentional.

---

## 3. Redundancy Detection Rules
A document is considered redundant if:
- its purpose overlaps more than 70% with another document
- its title implies the same structural role
- its content repeats core definitions already established
- it creates confusion in the meta or canopy layers
- it introduces parallel indexing without justification

Detection should be done during:
- coherence audits
- stability reports
- major expansions
- horizon revisions

---

## 4. Resolution Protocol
When redundancy is detected, follow these steps:

### Step 1 — Identify the Primary Document
Choose the version that:
- is most structurally aligned
- is most referenced by other files
- best fits UMA’s topology

### Step 2 — Classify the Redundancy
Is it:
- Structural?
- Plane-based?
- Narrative?

### Step 3 — Apply the Correct Action
- **Structural Redundancy:** merge or deprecate the secondary file  
- **Plane Redundancy:** move the secondary file to its correct plane  
- **Narrative Redundancy:** annotate the relationship in forest_roots.md  

### Step 4 — Update the Meta Index
Ensure the index reflects the resolved structure.

### Step 5 — Log the Change
Record the resolution in:
- forest_roots.md  
- stability_report.md  

This keeps the forest’s memory intact.

---

## 5. Redundancy Actions
UMA supports four actions:

### A. Merge
Combine content into the primary document.
Use when both documents contain essential material.

### B. Move
Relocate the secondary document to its correct plane.
Use when redundancy is caused by misplacement.

### C. Deprecate
Mark the secondary document as obsolete.
Use when it no longer serves a purpose.

### D. Parallelize
Allow both documents to exist intentionally.
Use when they serve different planes or interfaces.

---

## 6. Multi-Plane Considerations
Because UMA spans multiple conceptual planes:
- redundancy is expected
- parallel documents may be valid
- mythic and structural versions may coexist
- cartographic and meta versions may diverge

The protocol ensures these divergences remain coherent.

---

## 7. Horizon Interaction
Redundancy beyond the Horizon must be:
- explicitly labeled as external
- never structurally bound to UMA
- referenced only through controlled interfaces

The Horizon protects UMA from external index drift.

---

## 8. Closing Note
Redundancy is not a flaw.
It is a sign of growth, reflection, and multi-plane resonance.
This protocol ensures that resonance remains coherent, gentle, and aligned
with the forest’s topology.
