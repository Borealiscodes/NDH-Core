# 📄 **Collaboration Workflow Specification**  
*(Repo‑ready document)*

## 1. Purpose  
This specification defines the **collaboration workflows** for the Expanded Universe Engine.  
It ensures that multiple contributors (or future you) can safely:

- add modules  
- write case studies  
- perform analysis  
- cross‑map systems  
- refine boundaries  
- expand the engine  

without destabilizing the architecture or causing runaway emergence.

---

## 2. Collaboration Principles  
All collaboration follows six core principles:

- **Stability First**  
- **Bounded Expansion**  
- **Case‑Study‑Driven Development**  
- **Protocol Alignment**  
- **Safety Preservation**  
- **Cross‑Module Coherence**  

These principles ensure that collaboration strengthens the engine rather than fragmenting it.

---

## 3. Collaboration Workflow Overview  
The workflow consists of four phases:

1. **Preparation Phase**  
2. **Contribution Phase**  
3. **Integration Phase**  
4. **Verification Phase**

Each phase contains specific steps and boundaries.

---

## 4. Phase 1 — Preparation  
Before contributing, collaborators must:

### **4.1 Review Engine Boundaries**  
Ensures new work stays within defined scope.  
Guided Link: **Engine Boundaries**

### **4.2 Review Relevant Case Studies**  
Case studies define how the system behaves.  
Guided Link: **Case Study Corpus**

### **4.3 Identify Contribution Type**  
Contributions fall into one of the following categories:

- new module  
- new case study  
- analysis document  
- protocol refinement  
- pinned sequence  
- boundary adjustment  

This categorization determines the correct workflow path.

---

## 5. Phase 2 — Contribution  
Contributors follow structured steps depending on the type of addition.

### **5.1 Adding a New Module**  
Modules must include:

- purpose  
- boundaries  
- interaction rules  
- cross‑module dependencies  
- safety constraints  

Guided Link: **Module Specification**

---

### **5.2 Adding a Case Study**  
Case studies must:

- document emergent behavior  
- identify constraints  
- highlight interactions  
- avoid prescribing rules  
- remain observational  

Guided Link: **Case Study Protocol**

---

### **5.3 Adding an Analysis Document**  
Analysis must:

- reference existing case studies  
- identify patterns  
- avoid introducing new mechanics  
- remain descriptive, not prescriptive  

Guided Link: **Analysis Protocol**

---

### **5.4 Refining Protocols or Boundaries**  
Refinements must:

- be incremental  
- preserve stability  
- avoid expanding scope  
- be justified by case studies  

Guided Link: **Protocol Refinement**
