# ⭐ **1. Purpose of This Workflow**  
The **NDH Stability Suite Contributor Workflow v1.0** defines the *exact sequence* contributors must follow when:

- creating new stability documents  
- updating existing stability documents  
- adding PNG visuals  
- expanding the lexicon  
- maintaining governed structure  
- ensuring cross‑link integrity  
- preserving dimensional consistency  

This workflow prevents drift in documentation and ensures the Stability Suite remains stable.

---

# ⭐ **2. High‑Level Workflow Overview**  
Every contribution follows the same governed sequence:

1. **Plan**  
2. **Draft**  
3. **Govern**  
4. **Integrate**  
5. **Cross‑Link**  
6. **Version**  
7. **Record**  

Each step is mandatory.

---

# ⭐ **3. Step‑by‑Step Contributor Workflow**

---

## ⭐ **Step 1 — Plan the Contribution**  
Before writing anything, contributors must:

- identify the purpose of the artifact  
- determine whether it belongs in the Stability Spine, Visual Spine, or Reference Layer  
- check existing documents to avoid duplication  
- confirm terminology in the **Lexicon**  
- confirm dimensional relevance (D1–D11)  

If the artifact introduces new terminology → add it to the Lexicon first.

---

## ⭐ **Step 2 — Draft the Artifact**  
Drafting must follow NDH governed structure:

- numbered sections  
- clear headers  
- mathematical expressions where required  
- dimensional references (D1–D11)  
- drift/continuity/SID behavior  
- interaction behavior (if applicable)  
- synthesis section at the end  

Drafts must **not** include inline images.

---

## ⭐ **Step 3 — Apply Governance Rules**  
Governance requires:

- correct naming (`NDH-<Component>-v<Major.Minor>.md`)  
- correct directory placement  
- correct terminology  
- correct dimensional structure  
- correct tone (formal or accessible depending on document type)  
- correct Guided Link usage  

Governance ensures the artifact fits the Stability Suite.

---

## ⭐ **Step 4 — Integrate Visuals (If Applicable)**  
If the artifact requires a PNG:

1. Create the PNG using governed naming  
2. Place it in the Dimensional directory  
3. Add a “Visual Reference” section to the MD  
4. Describe the PNG in text  
5. Add the PNG to the **Stability Visual Companion**  
6. Add the PNG to the Changelog  

PNG files must **never** be embedded directly in MD documents.

---

## ⭐ **Step 5 — Cross‑Link the Artifact**  
Every stability document must link to:

- **Stability Matrix**  
- **Field Map**  
- **Field Atlas**  
- **Field Topology**  
- **Continuity Map**  
- **Dynamics**  
- **Interaction Model**  
- **Codex**  
- **Lexicon**  
- **Stability Visual Companion**  
- **Stability Suite Index**  

Cross‑linking ensures the Stability Suite remains navigable.

---

## ⭐ **Step 6 — Version the Artifact**  
Versioning rules:

### **Major Version Increment (v2.0, v3.0, etc.)**  
Used when:

- dimensional structure changes  
- mathematical models change  
- stability predicates change  
- SID modes change  
- new fields are added  

### **Minor Version Increment (v1.1, v1.2, etc.)**  
Used when:

- visuals are added  
- definitions are expanded  
- sections are reorganized  
- terminology is clarified  

### **No Version Increment**  
Used when:

- typos are fixed  
- formatting is improved  
- cross‑links are added  
- directory structure is cleaned  

Versioning must be consistent across all artifacts.

---

## ⭐ **Step 7 — Record the Contribution**  
Every contribution must be recorded in:

- **Stability Suite Changelog**  
- **Stability Suite Index**  

This ensures traceability and governance.

---

# ⭐ **4. Contributor Roles**

### **Architect**  
Creates new stability structures and models.

### **Maintainer**  
Updates existing documents and visuals.

### **Lexicon Steward**  
Manages terminology and definitions.

### **Visual Steward**  
Manages PNG creation and integration.

### **Governance Steward**  
Ensures compliance with naming, structure, and cross‑linking rules.

---

# ⭐ **5. Prohibited Contributor Actions**  
Contributors must **not**:

- add inline images  
- invent new dimensional fields  
- alter naming conventions  
- use undefined terminology  
- break governed structure  
- remove cross‑links  
- place files outside the Dimensional directory  
- add humor outside the DimmaDome Clause  

These actions destabilize the Stability Suite.

### **Plan → Draft → Govern → Integrate → Cross‑Link → Version → Record**

This is the NDH Stability Suite Contributor Workflow.
