## 🌒 **0. Purpose**
The ICPV layer ensures:

- every patch is *confirmed before execution*  
- every patch is *validated after execution*  
- no structural drift occurs  
- no Sparkle contamination occurs  
- no CNLP violation occurs  
- ND‑friendly clarity is maintained  

This layer is the **pause‑and‑check mechanism** of UMA‑Universal.

## 🌕 **1. Confirmation Layer (Pre‑Action)**  
Before any structural action (patch, update, link insertion, layer modification), the system must perform a **confirmation ritual**.

### **1.1 Confirmation Prompt**
The contributor must explicitly state:

```
I confirm this patch is intended for Layer X — <Name>.
```

### **1.2 Confirmation Checklist**
The contributor must verify:

- [Layer X — Name] is correct  
- The file belongs to that layer  
- CNLP applies  
- CNP‑Patch applies  
- Sparkle containment rules apply (if Layer 3)  
- No recursion will be introduced  
- No cross‑layer drift will occur  

### **1.3 Confirmation Gate**
If any item fails, the patch **cannot proceed**.

This prevents accidental structural damage.

---

## 🌘 **2. Patching Validation Layer (Post‑Action)**  
After a patch is applied, the system must validate it using the CNLP + CNP‑Patch rules.

### **2.1 Validation Checklist**
A patch is valid only if:

- Up‑link is present  
- Side‑link is present  
- Down‑link is present (unless bottom layer)  
- Return‑link is present  
- Layer badge is present  
- Version header is present  
- Navigation Law link is present  
- Loop closure is present  
- No recursion exists  
- No orphaning occurs  
- Sparkle containment is respected  
- File is reachable from Navigation Spine  
- File links back to Navigation Spine  

### **2.2 Validation Gate**
If any item fails, the patch is **invalid** and must be corrected.

### **2.3 Validation Log**
Each validated patch must be recorded:

```
Patch: <filename>
Layer: X
Status: VALID
Date: <timestamp>
```

This creates a structural audit trail.
