# 📘 **CNP‑Patch Protocol (Closed Navigation Patch Protocol v1.0)**  
*A structural patching standard for UMA‑Universal, ensuring CNLP compliance, ND‑friendly clarity, and Companion‑layer Sparkle containment.*

---

## **Purpose**
The CNP‑Patch Protocol defines the mandatory steps for patching any file within UMA‑Universal.  
It ensures:

- CNLP compliance  
- closed‑loop navigation  
- ND‑friendly structure  
- non‑recursive flow  
- Sparkle containment  
- contributor consistency  

This protocol governs **all structural updates**.

---

## **1. Pre‑Check (Mandatory Before Any Patch)**

Before patching any file, confirm:

### **1.1 CNLP is pinned at root**
```
/CNLP_Closed_Navigation_Loop_Protocol.md
```

### **1.2 Navigation Spine exists**
```
/NAVIGATION_SPINE.md
```

### **1.3 File layer is known and correct**
Every file must declare its layer via a badge:
```
[Layer X — Name]
```

If the layer is unclear, patching cannot proceed.

---

## **2. Apply Layer Badge + Version Header**

At the top of every patched file:

```
[Layer X — Name]
Version Y.Z — <Date>
```

This ensures provenance, clarity, and structural stability.

---

## **3. Apply CNLP Navigation Anchors**

Every patched file must include **all four CNLP anchors**:

### **3.1 Up‑Link (to Root Index)**
```
→ Integrated System Index
   /INDEX.md
```

### **3.2 Side‑Link (to Layer Index)**
Example for Companion layer:
```
→ Companion README Index
   /uma-core/Companions/README_Index.md
```

### **3.3 Down‑Link (to Next Layer)**
Example:
```
→ Documentation Architecture
   /uma-core/Companions/README_Index.md
```

### **3.4 Navigation Law**
```
→ Closed Navigation Loop Protocol
   /CNLP_Closed_Navigation_Loop_Protocol.md
```

These anchors ensure closed‑loop navigation and prevent orphaned files.

---

## **4. Apply Loop Closure**

At the bottom of every patched file:

```
← Back to Layer Index
```

This guarantees loop closure and ND‑friendly navigation.

---

## **5. Sparkle Containment Rules**

If the file contains Sparkle grammar:

### **5.1 It MUST be in Layer 3 — Companion**
Sparkle cannot appear in:

- RFC (Layer 0)  
- SID (Layer 1)  
- UMM (Layer 2)  
- CNLP  
- Structural Spine  
- Root README (except links)  

### **5.2 Sparkle files MUST include all CNLP anchors**
They must be fully loop‑integrated.

### **5.3 Sparkle grammar MUST remain non‑recursive**
No cross‑layer metaphors, no multi‑plane structures.

---

## **6. Validation Checklist**

A patch is valid when **all** of the following are true:

### **6.1 All four CNLP anchors are present**  
Up‑link, Side‑link, Down‑link, Return‑link.

### **6.2 Layer badge + version header are present**

### **6.3 Loop closure is present**

### **6.4 No recursion exists**

### **6.5 No Sparkle drift occurs**

### **6.6 File is reachable from Navigation Spine**

### **6.7 File links back to Navigation Spine**

If any condition fails, the patch is invalid.

---

## **7. Patch Flow Summary (ND‑Friendly)**

```
PRE‑CHECK
→ Apply Layer Badge
→ Apply CNLP Anchors
→ Apply Loop Closure
→ Enforce Sparkle Containment (if applicable)
→ Validate Patch
```

This is the complete patching workflow.
